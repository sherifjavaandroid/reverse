01

ما هو B(l)utter؟

التعريف

B(l)utter هو أداة Reverse Engineering مفتوحة المصدر متخصصة في تحليل تطبيقات Flutter المبنية على Android. تعمل الأداة عن طريق تجميع Dart AOT Runtime بنفس إصدار Dart المستخدم في التطبيق المستهدف، ثم استخدام هذا الـ Runtime لقراءة وفهم بنية الـ Snapshot الموجودة داخل libapp.so.

الأداة من تطوير worawit (المعروف أيضاً بـ sleepya)، وهو باحث أمني معروف ساهم في اكتشاف ثغرات كبيرة مثل EternalBlue. مكتوبة بـ C++20 مع scripts بـ Python.

💡
لماذا اسم B(l)utter؟ الاسم مزيج من Binary و Flutter - أداة لتحليل Flutter binaries. الأقواس حول (l) تشير إلى أن الـ "l" اختياري.

الميزات الرئيسية

  • ✓ استخراج أسماء الـ Classes والـ Methods والـ Fields من Dart AOT Snapshot
  • ✓ توليد Assembly مقروء مع رموز (symbols) كاملة
  • ✓ Dump كامل لـ Object Pool مع جميع الكائنات
  • ✓ تحليل الكود وتحديد أنواع المعاملات والقيم المرجعة
  • ✓ توليد Frida script جاهز للاستخدام مع template
  • ✓ توليد IDA Pro script لاستيراد الرموز
  • ✓ كشف تلقائي لإصدار Dart من libflutter.so
  • ✓ بناء تلقائي لـ Dart VM library المطلوب
  • ✓ دعم Compressed Pointers و Null Safety
02

لماذا نحتاج Blutter؟

مشكلة Flutter Reverse Engineering

تطبيقات Flutter تختلف جذرياً عن تطبيقات Android التقليدية:

Android التقليدي (Java/Kotlin) الكود يترجم إلى Dalvik bytecode في classes.dex. أدوات مثل jadx تستطيع عكسه بسهولة واستخراج كود Java/Kotlin شبه كامل.
Flutter (Dart) الكود يترجم مباشرة إلى ARM native code (AOT Compilation) داخل libapp.so. لا يوجد bytecode - فقط machine code خام يصعب تحليله.

التحديات بدون Blutter

التحديالوصفحل Blutter
لا رموز (Symbols) libapp.so لا يحتوي على أسماء الدوال أو الكلاسات في الحالة العادية يستخرج جميع الرموز من Dart Snapshot metadata
Snapshot Format خاص Dart يستخدم format مخصص لتخزين الكائنات والكود، يتغير مع كل إصدار يبني Dart VM بنفس الإصدار لقراءة الـ Snapshot بشكل صحيح
Object Pool معقد الثوابت والكائنات مخزنة في Object Pool بتنسيق ثنائي يعمل Dump كامل للـ Object Pool مع أنواع الكائنات وقيمها
Compressed Pointers Flutter يستخدم ضغط المؤشرات (32-bit بدلاً من 64-bit) مما يعقد التحليل يتعامل تلقائياً مع الـ Compressed Pointers
SSL Pinning معقد Flutter يستخدم BoringSSL مدمج في libflutter.so يوفر Frida template جاهز للـ hooking

المكونات الأساسية (C++ Source)

كود Blutter الأساسي مكتوب بـ C++20 ويتكون من المكونات التالية:

الملفالمسؤوليةالتفاصيل
main.cpp نقطة الدخول يحلل الـ arguments، يحمل libapp.so، يشغل التحليل والـ dump
DartApp النواة الرئيسية يحمل الـ ELF binary، يهيئ Dart VM، يقرأ Class Table و Object Pool، يبني خريطة Libraries/Classes/Functions
ElfHelper قراءة ELF يقرأ libapp.so كـ ELF file، يحدد مواقع الـ 4 snapshots: vm_snapshot_data, vm_snapshot_instructions, isolate_snapshot_data, isolate_snapshot_instructions
DartLoader تحميل Dart VM يهيئ Dart Isolate باستخدام snapshot data لقراءة metadata
DartLibrary مكتبات Dart يمثل كل library في التطبيق (مثل dart:core, package:flutter, etc)
DartClass الكلاسات يستخرج كل class مع الـ fields, methods, type arguments, inheritance, mixins
DartFunction الدوال يمثل كل function مع entry point, size, signature, parameters, closure info
DartStub الـ Stubs يتعرف على Dart VM stubs مثل AllocateObject, TypeCheck, MethodExtractor
DartTypes نظام الأنواع يدير Type, TypeParameter, FunctionType, RecordType (Dart 3+), TypeArguments
DartField الحقول يمثل حقول الكلاسات مع الـ offset, type, unboxed bitmap
CodeAnalyzer تحليل الكود يحول ARM64 assembly إلى IL (Intermediate Language)، يتتبع registers و local variables
Disassembler التفكيك يستخدم مكتبة Capstone لتفكيك ARM64 instructions
DartDumper التصدير يصدر Assembly files, Object Pool dump, Objects dump, IDA scripts
FridaWriter توليد Frida يولد Frida JavaScript script مع class definitions و helper functions

سير العمل الداخلي لـ DartApp

عندما يتم تشغيل Blutter، يحدث التالي داخل DartApp:

تحميل ELF Binary

ElfHelper::MapLibAppSo() يقرأ libapp.so ويحدد مواقع الـ 4 snapshot sections عبر البحث عن symbols مثل _kDartVmSnapshotData

تهيئة Dart VM

DartLoader::Load() يهيئ Dart Isolate باستخدام الـ snapshot data. هذا يسمح بالوصول إلى الـ metadata عبر Dart VM API

قراءة Class Table

loadFromClassTable() يمر على جميع الـ classes المسجلة في IsolateGroup، يستخرج الأسماء، الحقول، الـ methods، الـ inheritance chain

تحميل Stubs

loadStubs() يقرأ Object Store ويتعرف على جميع Dart VM stubs (مثل AllocateObject, StackOverflow, TypeCheck)

قراءة Object Pool

loadFromObjectPool() يمر على كل entry في الـ Object Pool ويربطها بالـ classes والـ functions المكتشفة

البحث عن Functions في Heap

findFunctionInHeap() يبحث عن Code objects في الـ heap لاكتشاف functions لم يتم ربطها بأي class (مثل closures مفقودة)

تحليل الكود (Code Analysis)

CodeAnalyzer::AnalyzeAll() يمر على كل function ويحول ARM64 assembly إلى IL مع تتبع registers وتحديد أنواع المعاملات

طريقة الاستخدام خطوة بخطوة

استخراج lib من APK

استخدم apktool أو unzip لاستخراج محتويات الـ APK. نحتاج مجلد lib/arm64-v8a/ الذي يحتوي على libapp.so و libflutter.so

# الطريقة 1: unzip مباشر unzip target_app.apk -d extracted_apk # الطريقة 2: apktool (يفك التشفير أيضاً) apktool d target_app.apk -o extracted_apk # الطريقة 3: مباشرة من APK (Blutter يدعم هذا) python3 blutter.py target_app.apk out_dir

تشغيل Blutter

الاستخدام الأساسي - أعطه مسار مجلد arm64-v8a ومجلد الإخراج:

# الاستخدام الأساسي python3 blutter.py extracted_apk/lib/arm64-v8a output # أو مباشرة من APK python3 blutter.py target_app.apk output

أول تشغيل (البناء التلقائي)

عند أول تشغيل لإصدار Dart جديد، Blutter سيقوم تلقائياً بـ:

1. كشف إصدار Dart يقرأ libflutter.so لاستخراج engine ID وإصدار Dart
2. تنزيل Dart SDK source Git sparse checkout لـ runtime و tools فقط (يوفر حجم التنزيل)
3. بناء Dart VM library يبني static library بـ cmake + ninja
4. بناء Blutter executable يربط Blutter مع Dart VM library المبني
💡
أول تشغيل قد يستغرق 10-30 دقيقة حسب سرعة الإنترنت والجهاز. التشغيلات اللاحقة لنفس إصدار Dart ستكون سريعة جداً لأن الـ executable محفوظ في bin/.

07

فهم ملفات المخرجات

asm/* - Assembly Files

مجلد يحتوي ملفات Assembly لكل library في التطبيق، منظمة بنفس هيكل packages التطبيق.

كل ملف يحتوي:

  • أسماء الـ Classes مع الـ inheritance
  • أسماء الـ Methods مع الـ parameters
  • ARM64 Assembly مع رموز واضحة
  • تعليقات على Object Pool references
  • تعليقات على Thread offsets
// class LoginService extends Object { // _ login(String username, String password) { 0x1a2b4c: stp fp, lr, [sp, #-0x10]! 0x1a2b50: mov fp, sp 0x1a2b54: ldr x0, [PP, #0x1f8] ; "https://api.example.com/login" 0x1a2b58: bl HttpClient.post // } // }

pp.txt - Object Pool

Dump لكل كائن في Object Pool مع الـ offset والنوع والقيمة.

هذا الملف ذهبي لأنه يحتوي:

  • جميع الـ String constants (URLs, API keys, etc)
  • جميع الـ Integer/Double constants
  • مراجع الـ Classes و Functions
  • Type objects و TypeArguments
  • Closures و Field references
# Object Pool Entries [pp+0x0048] Obj!String "Authorization" [pp+0x0050] Obj!String "Bearer " [pp+0x0058] Obj!String "https://api.example.com" [pp+0x0060] Obj!Type int [pp+0x0068] Obj!Function LoginService.login [pp+0x0070] Obj!Class UserModel

objs.txt - Objects Dump

Dump تفصيلي (nested) لكل Object مع حقوله وقيمه المتداخلة. يعرض الكائنات بشكل هرمي.

# Nested Object Dump Obj!_AppConfig { baseUrl: "https://api.example.com" apiKey: "sk_live_abc123..." timeout: 30000 headers: Map { "Content-Type": "application/json" "X-App-Version": "2.1.0" } }

blutter_frida.js - Frida Script

Script Frida جاهز مع:

  • جميع Class IDs و sizes
  • Field offsets لكل class
  • Helper functions لقراءة Dart objects
  • دعم Compressed Pointers
  • Template جاهز للـ hooking

ida_script/ - IDA Pro Scripts

مجلد يحتوي scripts لـ IDA Pro لاستيراد جميع الرموز المكتشفة. يتضمن:

  • أسماء الدوال مع عناوينها
  • Struct definitions للـ classes
  • تعريفات الأنواع
08

تحليل الكود المصدري التفصيلي

main.cpp - نقطة الدخول

الملف الرئيسي الذي يربط كل المكونات. تسلسل العمل:

// 1. إنشاء DartApp وتحميل libapp.so DartApp app{ libappPath.c_str() }; // 2. الدخول في Dart scope وتحميل المعلومات app.EnterScope(); app.LoadInfo(); // يقرأ Class Table + Object Pool + Functions app.ExitScope(); // 3. تحليل الكود (ARM64 -> IL) app.EnterScope(); CodeAnalyzer analyzer{ app }; analyzer.AnalyzeAll(); // يحلل كل function // 4. تصدير النتائج DartDumper dumper{ app }; dumper.DumpObjectPool("pp.txt"); // Object Pool dumper.DumpObjects("objs.txt"); // Objects nested dump dumper.DumpCode("asm/"); // Assembly with symbols dumper.Dump4Ida("ida_script/"); // IDA Pro scripts // 5. توليد Frida script FridaWriter fwriter{ app }; fwriter.Create("blutter_frida.js");

DartApp - النواة (الأهم)

هذا هو القلب النابض لـ Blutter. يحتوي على:

البيانات الداخلية الرئيسية:
  • lib_base - عنوان قاعدة تحميل libapp.so
  • heap_base_ - عنوان بداية Dart Heap
  • ppool - مؤشر إلى Object Pool
  • libs - قائمة جميع المكتبات (vector<DartLibrary*>)
  • classes - قائمة جميع الـ classes بالـ CID (vector<DartClass*>)
  • functions - خريطة من address إلى function (unordered_map)
  • stubs - خريطة من address إلى stub
  • staticFields - خريطة الحقول الثابتة
  • typeDb - قاعدة بيانات الأنواع

الـ CIDs (Class IDs) المدمجة التي يتتبعها:

dartIntCid - Integer
dartDoubleCid - Double
dartStringCid - String
dartBoolCid - Bool
dartListCid - List
dartMapCid - Map
dartSetCid - Set
dartFutureCid - Future
dartRecordCid - Record (3.0+)

CodeAnalyzer - تحليل ARM64

يحول ARM64 machine code إلى تمثيل وسيط (IL) قابل للفهم:

AnalyzedFnData يحتوي:
  • asmTexts - ARM64 instructions مع annotations
  • stackSize - حجم الـ stack frame
  • params - المعاملات المكتشفة (FnParams)
  • il_insns - الـ IL instructions المولدة
  • returnType - نوع القيمة المرجعة
  • state - حالة الـ registers والـ local variables

كل AsmText يحمل معلومات إضافية:

DataTypeالوصف
ThreadOffsetوصول إلى Thread object (مثل Stack limit, Top of heap)
PoolOffsetوصول إلى Object Pool entry (ثابت أو reference)
Booleanقيمة boolean
Callاستدعاء function أو stub

Frida Script المولد

هيكل blutter_frida.js

الـ script المولد يحتوي على ثلاثة أقسام رئيسية:

1. Template Section (ثابت) يأتي من scripts/frida.template.js ويحتوي:
  • tryLoadLibapp() - ينتظر تحميل libapp.so
  • init(context) - يقرأ HeapAddress من register x28
  • getArg(context, idx) - يقرأ argument من الـ stack
  • getTaggedObjectValue() - يحول tagged pointer إلى قيمة
  • getObjectValue() - يقرأ قيمة object حسب CID
  • getDartString/Bool/Int/Double/Array() - قراءة أنواع محددة
  • decompressPointer() - فك ضغط compressed pointer
2. Classes Section (مولد) يحتوي تعريفات كل class مع:
  • id - Class ID
  • sid - Super class ID
  • size - حجم الـ instance
  • argOffset - موقع type arguments
  • fbitmap - Unboxed fields bitmap
  • valOffset/lenOffset/dataOffset - لأنواع مثل String, Array
3. Hook Template مثال جاهز للاستخدام في onLibappLoaded():
// مثال: Hook على function معينة function onLibappLoaded() { // عنوان الدالة من asm/ output const fn_addr = 0x1a2b4c; Interceptor.attach(libapp.add(fn_addr), { onEnter: function() { // تهيئة HeapAddress init(this.context); // قراءة أول argument let objPtr = getArg(this.context, 0); // تحويله إلى قيمة قابلة للقراءة const [tptr, cls, values] = getTaggedObjectValue(objPtr); console.log(`${cls.name} =`, JSON.stringify(values, null, 2)); } }); }

Dart Object Memory Layout

فهم كيف يخزن Dart الكائنات في الذاكرة (ضروري لـ Frida):

// Tagged Object Pointer (Compressed - 4 bytes) // Bit 0 = 1 → Heap Object (subtract 1 to get real pointer) // Bit 0 = 0 → Smi (shift right 1 to get value) // Object Header (first 8 bytes): // Bits [12:23] → Class ID (CID) // Bits [8:11] → Size tag (0 = variable size) // Bits [32:63] → Hash / Identity // Compressed Pointer Decompression: // real_ptr = heap_base + compressed_ptr (sign-extended 32-bit) // heap_base = Thread.x28 << 32 // String Layout: // [header:8] [length_smi:4] [hash:4] [data:N] // length = length_smi >> 1 (Smi encoding) // Array Layout: // [header:8] [type_args:4] [length_smi:4] [element_0:4] [element_1:4] ...

التعامل مع Obfuscated Apps

عندما يستخدم المطور --obfuscate flag:

  • أسماء الـ classes و functions تصبح عشوائية (مثل aB, cD)
  • لكن Blutter لا يزال يستطيع استخراج:
  • - هيكل الـ classes (fields, methods, inheritance)
  • - String constants في Object Pool (URLs, keys)
  • - Flow الكود والـ call graph
  • - أسماء الـ Dart SDK classes (لا تتأثر بالـ obfuscation)
💡
الـ obfuscation في Flutter يغير فقط أسماء الـ user code. أسماء مثل HttpClient, SharedPreferences, setState تبقى كما هي لأنها من SDK.

حل المشاكل الشائعة

❌ خطأ: "Cannot find libapp file" ▶

تأكد أنك تمرر المسار الصحيح لمجلد arm64-v8a الذي يحتوي على libapp.so و libflutter.so.

# صحيح: python3 blutter.py extracted/lib/arm64-v8a output # خطأ: python3 blutter.py extracted/lib/ output # ناقص arm64-v8a
❌ خطأ في البناء على Windows ▶

تأكد أنك تستخدم "x64 Native Tools Command Prompt for VS" وليس CMD العادي. وتأكد من تشغيل python scripts\init_env_win.py أولاً.

❌ إصدار Dart غير مدعوم ▶

جرب --no-analysis للإصدارات القديمة، أو git pull ثم --rebuild لتحديث Blutter:

git pull python3 blutter.py path/to/arm64-v8a output --rebuild
❌ البناء يستغرق وقتاً طويلاً ▶

أول بناء لإصدار Dart جديد طبيعي أن يستغرق 10-30 دقيقة. المشاريع اللاحقة لنفس الإصدار ستكون فورية. يمكنك حذف build/ و dartsdk/ بعد البناء لتوفير المساحة.
